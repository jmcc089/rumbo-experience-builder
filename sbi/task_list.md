- **Public service surface** (`project/src/lib/booking/index.ts`), for client-portal SBIs (09/10):
  - `createRequest(intake: IntakeInput): Promise<{ id: string; token: string }>` — persists only, status `building`.
  - `runRequestPipeline(requestId: string, hooks?: PipelineHooks): Promise<void>` — match filter → availability confirm → LLM extraction/weights → `assemble()` → stores proposals, sets status `proposals_ready`. `hooks.notifyProposalsReady` is the SBI-08 email-2 wiring point (currently a no-op if omitted).
  - `getProposals(token: string): Promise<ProposalsView>` — `{ status: 'ready'|'expired'|'not_ready'|'not_found', proposals?, expiresAt? }`. Starts the 15-min hold on first call only (idempotent on repeat calls).

  - `HOLD_WINDOW_MINUTES` exported constant (15).
- **Match filter** (`pipeline.ts`): loads all experiences/providers/lodging (small dataset, no pagination needed), filters by `interests` (dropdown categories, skipped if empty), `open_days` overlapping any day-of-week in the trip range, and a loose per-item sanity check (`applyMarkup(net_price) <= budgetTotal`). Full lodging pool is passed through uncut — tier weighting happens inside the engine's scoring, not here.
- **Availability confirmation**: resolved **once per experience per request**, using the arrival date + the experience's own `open_from` as the representative slot (SBI-04's confirmation model is per-slot, not per-trip-day; this stands in for "the provider portal was asked and answered"). Confirmed experiences become `CandidateExperience[]` fed to `assemble()`.
- **Real-order consumption wiring**: `spotsConsumedByRealOrders(experienceId, date)` in `consumption.ts` joins `order_items → orders → client_requests`, computes each item's absolute calendar date as `request.arrival_date + (day_index - 1)`, and sums `travelers` for paid, non-disrupted items matching the target date. This is the concrete implementation of SBI-04's stubbed `realOrderConsumption` term.
- **Known gotcha (fixed during build):** `@neondatabase/serverless` returns Postgres `date` columns as JS `Date` objects, not `'YYYY-MM-DD'` strings — silently breaks any string-templated date math (`${date}T00:00:00Z`) with `Invalid Date`. `store.ts` now has a `toDateString()` normalizer applied to every `date` column read from the DB (`arrival_date`, `departure_date`). Any future module reading date columns directly via `pool.query` must normalize the same way — `time` columns come back as strings already, no issue there.
- **Order materialization**: one `order_items` row per lodging-per-day (`net_price` = `net_price_per_night` looked up from catalog) and one row per scheduled experience (`net_price` = catalog `net_price` × `travelers`, i.e. total booked cost, not per-person). Insert is wrapped in a single DB transaction (`BEGIN`/`COMMIT`/`ROLLBACK`) alongside setting `client_requests.status = 'paid'`.
- **Personalization/provider-instructions timing — deviation from the SBI's literal step order:** the SBI text lists "engine assemble → personalization" inside `runPipeline`, but SBI-06's own handoff notes say instructions are "attached... on the order" — i.e. only for the *one chosen* itinerary's providers, which isn't known until `confirmAndPay`. Generating instructions for all 3 proposals' providers upfront would be wasteful and premature. Implemented as: `extractConstraints` (personalization_notes + hard constraints) runs in `runPipeline` and is persisted on `client_requests.extraction_json`; `generateProviderInstructions` is called in `confirmAndPay`, once per distinct provider used in the chosen itinerary, using the persisted notes. Logged here per the orchestrator's "note deviations rather than improvise silently" rule — flag if a human reviewer wants it moved back into the pipeline step.
- **Not wired (documented, not a bug):** `extraction.extra_hard_constraints` (dietary/mobility) are additive filters per SBI-06, but the current catalog schema (`experiences`, `lodging`) has no per-item dietary/mobility field to filter against — only `provider_personalization.dietary_options` (free text, used for personalization copy, not machine-checkable). No candidate is currently dropped by this filter; the wiring point is commented in `pipeline.ts`. Logged in Open issues below too.
- Verification (`verify.ts`) passed all checks against the live Neon DB: unique/non-guessable tokens, hold starts only on first `getProposals` (idempotent on repeat reads), pay-within-window materializes `orders`+`order_items` and flips status to `paid`, a paid order measurably reduces `spotsConsumedByRealOrders` for a second lookup, forced expiry (`expires_at` set to the past) correctly reports `expired` from both `getProposals` and `confirmAndPay` and flips request status to `expired`, and provider net price never appears in the client-facing itinerary payload. `npm run build` / `tsc --noEmit` clean.

### SBI-13 notes
- **Files created:** `project/src/lib/repair/{store.ts,index.ts,verify.ts}`. Verification: `npx tsx src/lib/repair/verify.ts` (run from `project/` — hits the live Neon DB against real seed data; creates + cleans up its own paid orders each run, no manual purge needed).
- **Reused from SBI-07 (exported for this purpose):** `matchFilter`, `confirmAvailability`, `loadTransferMatrix`, `tripDates` in `project/src/lib/booking/pipeline.ts` were changed from module-private to `export` so repair can rebuild a candidate pool for the gap day without duplicating the match/availability logic. No behavior change to `runPipeline` itself.
- **Disruption generator** (`repair/index.ts::disruptOrder(orderId)`): picks one `order_items` row (`item_type='experience'`, `status='booked'`) weighted by `Math.max(1 - provider.reliability_score, 0.01)` — low-reliability (informal) providers are more likely to be picked, but every provider has a nonzero floor chance. The random draw is `hashToUnitFloat(orderId, "disruption")` (SBI-04's FNV-1a hash helper) — deterministic per order, not `Math.random()`. Marks the picked row `status='disrupted'`. Returns `{ disrupted, orderItemId, experienceId, dayIndex, providerId, reliabilityScore }` or `{ disrupted:false, reason }` if there are no bookable experience items left.
- **Repair invocation** (`repair/index.ts::repairOrder(orderId)`): finds the (first) `order_items` row with `status='disrupted'`, rebuilds a fresh `RepairProblem` for that gap day only (candidate pool re-matched + re-confirmed for the gap day's calendar date, explicitly excluding the disrupted experience id so it can't be re-picked as its own replacement), and calls the SBI-05 engine's `repair()`. Deterministic weights are re-derived from the request's stored `prefs_json` via `deriveWeights`/`selectedInterests` (same LLM-boundary helpers SBI-07 uses) — the LLM itself is never called here.
- **Price differences on replacement: reflected, not absorbed.** On success, `store.ts::applyRepair` marks every existing `experience` order_item for that `day_index` as `replaced` (not just the one disrupted piece — the engine's `repair()` regenerates the whole gap day, so all of that day's experiences are superseded together), inserts fresh `booked` rows for the new day's experiences, and overwrites `orders.chosen_itinerary_json` + `orders.client_price` with the engine's recomputed `ItinerarySnapshot`/`client_total`. The client-facing total can go up or down after a repair — this is intentional and visible in the order record, not silently absorbed.
- **Failure surfacing:** if `repair()` finds no valid replacement (empty candidate pool, no schedulable window, etc.), `repairOrder` returns `{ repaired:false, reason }` straight from the engine's `RepairResult.reason` and the item stays `disrupted` (no partial DB writes). Confirmed rare in practice given seed redundancy per SBI-03/13's own criteria — not specifically forced in verify.ts since a real unsolvable case wasn't reproducible with the current seed within a single trip, so verify.ts instead exercises "no disrupted item to repair" and "order not found" as the two guaranteed clean-failure paths.
- **Demo trigger surface: operator dashboard, not a background job.** New "Repair demo" section on `/operator` (`project/src/app/operator/{actions.ts,RepairPanel.tsx}`, `page.tsx` wired to fetch `getOrdersForRepair()` from the new `project/src/lib/operator/store.ts::getOrdersForRepair` query). Each paid order gets a row with booked/disrupted/replaced counts and two buttons: **Simulate disruption** (`disruptOrder`) and **Repair** (`repairOrder`, disabled until `disrupted_count > 0`), both `"use server"` actions in `operator/actions.ts` that `revalidatePath("/operator")` on completion. This is explicit and click-triggered per the SBI's "not a daemon" instruction.
- **Verified (live Neon + browser):** `verify.ts` — disruption is deterministic and marks exactly one `booked→disrupted` item; a second disruption on a fresh twin order doesn't re-pick the same item; repair returns a valid replacement, marks the old gap-day item(s) `replaced`, inserts new `booked` experience item(s), and `orders.client_price` matches the engine's recomputed total; repairing again with nothing disrupted fails cleanly with a reason; repairing a nonexistent order fails cleanly.
